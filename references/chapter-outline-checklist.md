# 章纲自检清单

章纲（chapter.yaml）是本章写作的唯一需求文档——subagent 拿到它就去写正文，不再看前文、不再问意图。章纲里没写的，正文里就不会出现；章纲里写糊的，正文里一定跑偏。**章纲是 subagent 的眼睛、耳朵和判断力。它看不懂的事，你写再多字也白搭。**

> **两条底线**（任一不过，禁止进入提示词生成阶段（3.3））：
>
> 1. **每条 memo 可执行**——subagent 读完后能说出"我要在正文的第 X 段写 Y 这件事"。如果 subagent 只能说出"这一章要有悬念感"，memo 等于没写。
> 2. **章纲不含上帝视角解构**——outline 和 memo 中不得出现"本章讲述了""主角经历了""通过 X 展现了 Y"等分析性语言。这些句子被 subagent 抄进正文的概率极高，一旦抄进去，整章报废。

---

## 最高原则：章纲是写给 subagent 的需求文档，不是写给作者的笔记

> 下面所有规则之前，先问一句：**"一个没有直觉、没读过前文的 subagent，拿着这份章纲和提示词，能写出正确的正文吗？"**

作者知道"主角经历了什么"，但 subagent 不知道。作者知道"这一章要有悬念感"是什么意思，但 subagent 只看到六个字。作者知道"陆征发现线索"具体是翻抽屉还是查数据库，但 subagent 不知道——它会在两者之间随机选，或者两样都写。

**判定标准：**
- memo 的每个字段，替换成"这件事发生在正文的哪个位置？以什么方式呈现？"——答不出来就是没写清楚
- outline 的每一条 key_point，替换成"subagent 能用它写出正文的一个段落吗？"——写不出来就是太模糊
- 整份章纲读完后，可以在脑子里过一遍本章的完整正文——过不下来就是有缺口

**允许的：**

| 行为 | 说明 |
|------|------|
| memo 中写具体可写的场景指令（如"在出租屋床垫下发现一张对折的纸条"） | 好——subagent 知道写什么 |
| 部分 memo 段留空（如本章没有过渡场景，downtime_functions 可为空） | 但留空必须有理由，不是偷懒 |
| outline 的 key_points 比卷纲的 summary 更具体 | 这是章纲的功能——把卷纲的"勘查现场"展开为"进入出租屋→发现床没铺→发现充电器→找到便利贴" |

**禁止的：**

| 行为 | 后果 |
|------|------|
| memo 里写抽象形容词评价（"本章节奏紧凑""这一章写得很精彩"） | subagent 看不懂，直接忽略 |
| memo 里写"同上""参见前章" | subagent 没有前章的内存——每一章的 memo 必须自足 |
| 拿卷纲的 summary 直接当章纲的 memo.current_task | 卷纲是定位，章纲是指令——级别不同，不能套用 |
| memo 7 段中任何一段留空无理由 | 表示那一段的情绪设计没有想清楚 |

---

## 一、memo 7 段

memo 是章纲的核心——7 段分别对应 subagent 写作时需要的 7 种指令。每一段都要回答一个不同的"怎么写"的问题。

**每段自检的通用标准：**

| 标准 | 判定方法 |
|------|---------|
| 可执行 | 读完后能说出"正文的哪一段要写什么" |
| 无抽象形容词 | 没有"好""不错""精彩""紧张""有趣"等空洞评价词 |
| 无分析性语言 | 没有"通过X展现了Y""这标志着""这意味着""完成了XX的转折" |
| 具体到事件/动作/物品 | 写的是"什么事发生"而非"什么主题被表达" |

### 1. current_task（当前任务）

subagent 需要知道"这一章的目标是什么"——不是主题，不是感受，是完成什么具体动作。

| 字段 | 填写标准 | 常见错误 |
|------|---------|---------|
| current_task | 一句话：本章要完成的具体动作 + 与哪条故事线的什么关系。读完后 subagent 知道整章的航向 | 写成主题（"本章讲信任"）、写成感受（"本章要有悬念"）、写成卷纲 summary 的复读 |

```
✅ 合格：
"建立陆征的人物形象和日常工作状态，通过苏沫的委托引入主线案件"
→ subagent 知道：要写日常状态 + 要写委托场景 + 要引入案件

"通过现场勘查推进案件调查，线索逐渐指向苏棠不是主动离开的"
→ subagent 知道：要写勘查 + 方向是推翻警方结论

"将苏棠失踪案与三年前旧案建立联系，揭示案件远比失踪更复杂"
→ subagent 知道：要写旧案关联 + 结果是案件升维

❌ 不合格：
"本章建立悬念"
→ 建立什么悬念？通过什么事建立？subagent 只能猜

"主角继续调查案件"
→ "继续"是什么意思？调查什么？发现了什么？等于没写

"本章节奏紧凑，推进主线"
→ 全是评价词——subagent 不知道要写什么
```

### 2. reader_expectation（读者期待管理）

这是整份 memo 的核心——决定了本章的情绪走向。必须告诉 subagent：读者在上一章结束时处于什么状态，本章该怎么对待这个状态。

| 字段 | 填写标准 | 常见错误 |
|------|---------|---------|
| state | 精确的情绪词（好奇/紧张/满足后期待/愤怒/被压制后期待翻盘/空白）。不是泛泛的"期待" | 写"期待"——期待什么？是等真相还是等对决？完全不同的写作策略 |
| strategy | 本章对读者情绪缺口的处理方式：制造新缺口 / 延迟兑现 / 兑现旧缺口 / 双重（先兑现再制造） | 写"维持悬念"——维持什么悬念？怎么维持？ |
| detail | 具体描述读者在等什么：等谁出场/等某个真相/等一场对决/等关系突破。必须具体到人/事/物 | 写"读者想知道接下来会发生什么"——等于没写 |

```
✅ 合格：
state: "好奇——上一章结尾留下悬念，读者想知道苏棠到底去哪了"
strategy: "制造新缺口"
detail: "读者期待调查有进展——给他们一些进展（房间矛盾细节、被删聊天记录），同时制造新的疑问（楼下的男人是谁？）"
→ subagent 知道：前半章给进展，后半章埋新钩子。情绪的"先给后要"节奏清晰

state: "紧张中带好奇——上章结尾被人盯上，读者担心陆征的安全，同时想知道苏棠的下落"
strategy: "制造新缺口"
detail: "读者在等调查进展和苏棠下落——给一部分进展（旧案联系），但同时告诉他们更大的麻烦在后面（办公室被撬）"
→ subagent 知道：进展要给但不能给满，章末必须升级威胁

❌ 不合格：
state: "期待"
strategy: "维持悬念"
detail: "读者期待下一章的内容"
→ 三行全是空词——subagent 不知道读者在等什么、该给什么、该压什么

state: "好奇"
strategy: "制造新缺口"
detail: "本章要制造悬念"
→ 制造什么悬念？通过什么事件制造？subagent 只能瞎编
```

### 3. payoff_plan（伏笔兑现计划）

告诉 subagent 本章该碰哪些钩子、不该碰哪些钩子。钩子操作必须具体到触发条件和推进方式。

| 字段 | 填写标准 | 常见错误 |
|------|---------|---------|
| must_resolve | 本章必须兑现的钩子 ID 列表。空时表示本章不收束任何旧钩子——必须有理由 | 有钩子该收了但列表仍然为空 |
| must_hold | 本章绝对不能碰的底牌 ID 列表。每条必须有理由：为什么现在不能掀 | 空列表——vol 1 ch 1 可以，但中后期通常有压住的底牌 |
| partial_advance | 本章推进但不完全兑现的钩子列表。每条必须写推进方式（不是"推进了hook-001"——要写怎么推进） | 只写 hook_id 不写推进方式——subagent 不知道该推进多少 |

```
✅ 合格：
must_resolve: []  # 第 1 章，没有要收的旧钩子
must_hold: ["hook-002"]  # 陆征离开警队的原因还不能揭示
partial_advance:
  - id: "hook-001"
    advance: "种下主线钩子——苏棠失踪，警方判定主动离开，但细节有矛盾"
→ subagent 知道：本章要埋 hook-001，不碰 hook-002

must_resolve: []
must_hold: ["hook-002"]
partial_advance:
  - id: "hook-001"
    advance: "发现苏棠的失踪与三年前旧案的证人有关联——案件升维，从失踪案变为可能涉及灭口的刑事案件"
  - id: "hook-002"
    advance: "通过方岩的对话隐约暗示陆征三年前离开警队的原因——与那个被压下来的案子有关"
→ subagent 知道：hook-001 要推进到"案件升维"，hook-002 只能"隐约暗示"——推进边界清晰

❌ 不合格：
partial_advance:
  - id: "hook-001"
    advance: "推进主线钩子"
→ subagent 不知道"推进"是什么意思——可能推一毫米，也可能直接收束

must_resolve: ["hook-003"]
must_hold: ["hook-003"]
→ 同一条钩子既要收又要压——逻辑矛盾，说明没想清楚
```

### 4. downtime_functions（过渡/日常场景功能映射）

非冲突段落不是"休息"——每段过渡场景必须有明确的叙事功能。如果一段场景没有功能，它就不该存在。

| 字段 | 填写标准 | 常见错误 |
|------|---------|---------|
| downtime_functions | 每条格式：`[段落位置/场景] → [承担功能]`。功能必须是具体动作（埋伏笔/推进关系/建立反差/信息铺垫/情绪缓冲），不是空描述 | 写"日常场景"或"过渡段落"——没有标注功能，subagent 会写无意义填充 |

```
✅ 合格：
- "开头办公室场景 → 建立陆征的日常状态和性格底色"
  → subagent 知道：开头场景的功能是展示人物，不是随便写写氛围
- "陆征观察苏沫的细节 → 展示陆征的职业习惯和观察力"
  → subagent 知道：写观察细节是为了建立专业形象，不是为了凑字数
- "陆征去物流公司路上的环境描写 → 南岸新区与老城区的对比，暗示城市的两面性"
  → subagent 知道：环境描写有叙事功能（暗示城市暗面），不是纯风景

❌ 不合格：
- "过渡段落"
  → 没有功能映射——subagent 不知道这段要干什么，会写无意义填充
- "日常描写"
  → subagent 不知道日常描写承担什么功能，可能写一页吃饭睡觉
- "环境渲染氛围"
  → 氛围服务于什么？不知道——subagent 可能堆砌比喻
```

### 5. key_choices（关键抉择三连问）

本章中角色的重要选择必须回答三个问题：为什么做、符合人设吗、读者会觉得突兀吗。

| 字段 | 填写标准 | 常见错误 |
|------|---------|---------|
| key_choices | 每条包含：谁做了什么选择 + 为什么（利益/情感驱动）+ 为什么符合人设（过往经历+当前利益+性格底色）+ 读者感受验证 | 只写"主角做了选择"不写原因——subagent 不知道角色动机，会在正文里编造理由 |

```
✅ 合格：
- "陆征为什么接下案子？——一半是需要收入维持事务所运转，一半是苏沫提到的朋友圈地址查无此址让他觉得警方确实草率了。符合他的人设：不急于下判断，相信细节。"
  → subagent 知道：不是热血也不是同情，是专业警觉+现实需要。正文里陆征的行为动机有据可查

- "陆征为什么不直接告诉方岩全部信息？——他不确定方岩在这个案子里扮演什么角色，三年前的事让他对任何警局的人都有保留。这是性格驱动的选择。"
  → subagent 知道：对话中的保留不是因为情节需要，而是因为角色性格——写对话时就有分寸感

- "方岩为什么要私下警告陆征？——他对陆征有愧疚，但也不希望陆征捅出大篓子连累自己。矛盾的选择符合他的人设。"
  → subagent 知道：方岩的行为是矛盾而真实的——有愧疚也有自保，不是单纯的"好人帮忙"

❌ 不合格：
- "主角选择了调查"
  → 为什么调查？什么驱动？符合人设吗？全部没答——subagent 会默认写成"因为他正义"
- "反派决定阻止主角"
  → 什么方式阻止？什么动机？读者会觉得反派"蠢"还是"有理由"？没想清楚
```

### 6. required_changes（章尾必须发生的改变）

这是 subagent 最重要的航标——本章结束时，什么东西必须变了。如果章末什么都没变，这章就不该存在。

| 字段 | 填写标准 | 常见错误 |
|------|---------|---------|
| required_changes | 1-3 条具体改变，每条标注改变类型（信息/关系/物理/权力）+ 从什么变成什么。不是"有进展"——是"什么具体事情变了" | 写"有进展"——什么进展？谁知道了什么？写"关系变化"——谁和谁？从什么变成什么？ |

```
✅ 合格：
- "信息改变：陆征（和读者）知道了苏棠失踪的基本事实和几个疑点"
  → subagent 知道：章末读者必须知道失踪的基本情况+疑点。这是信息改变的具体内容
- "信息改变：读者和陆征知道苏棠不是主动离开的——房间状态不符合计划出行的特征"
  → subagent 知道：改变不是"调查推进了"，是"确认了非主动离开"——具体的知识增量
- "威胁引入：楼下的男人盯着陆征——调查被人盯上了"
  → subagent 知道：必须写一个被盯上的场景，而且读者必须感知到威胁
- "物理改变：陆征的办公室被侵入——他的安全空间不再安全"
  → subagent 知道：要写办公室被侵入的发现过程，以及这个改变带来的心理冲击

❌ 不合格：
- "调查有进展"
  → 什么进展？知道了什么新东西？没有新信息 = 没有改变
- "关系变化"
  → 谁和谁的关系？从什么变成什么？是变好还是变坏？全空
- "故事推进"
  → 空词——每章都在推进故事，关键是什么被推进了
```

### 7. prohibitions（硬约束红线）

告诉 subagent 绝对不能做什么。不是"最好不要"，是"做了就报废"。

| 字段 | 填写标准 | 常见错误 |
|------|---------|---------|
| prohibitions | 每条是具体的禁止场景或行为。格式："不要在 X 场景中做 Y 事"或"不要让 X 角色做 Y 事"。必须可验证——正文写完后能对照检查 | 写抽象禁忌（"不要太拖沓""不要太紧张"）——subagent 无法执行 |

```
✅ 合格：
- "不要揭示陆征三年前离开警队的详细原因——只通过细节暗示他以前是刑警"
  → subagent 知道：可以暗示但不能说破。正文写完后可以逐句扫描"三年前""离开警队"等关键词
- "不要让楼下的男人和陆征对话——只通过眼神和氛围制造压力"
  → subagent 知道：肢体可写、氛围可写，但对话是红线
- "不要揭示聊天记录的发件人身份"
  → subagent 知道：聊天记录的内容可以写，但谁发的不能揭示
- "不要在章末揭示跟踪者的身份"
  → subagent 知道：跟踪行为可以写，但跟踪者是谁不能写

❌ 不合格：
- "不要太拖沓"
  → 怎么算拖沓？多少字算拖沓？subagent 没有判断标准
- "保持紧张感"
  → 这不是禁止，是期望。prohibitions 是"不要做什么"，不是"要做什么"
- "不要写得太无聊"
  → 无法验证——对 subagent 来说，它写的所有东西都不无聊
```

## 二、emotional_design（情绪设计）

情绪设计不是给作者看的情绪调色盘——是给 subagent 的情绪操作指令。subagent 需要知道从哪个情绪开始、经过什么情绪转折、以什么情绪结束、高潮在哪一段。

### 1. 情绪起点/落点

| 字段 | 填写标准 | 常见错误 |
|------|---------|---------|
| primary_mood | 一个精确的情绪词（好奇/紧张/压抑/爽快/温情/悲伤/愤怒/释然）。不是"复杂""多元"——subagent 需要一个主色调 | 写"复杂""混合"——subagent 不知道以哪个为主，会写得情绪模糊 |
| mood_progression | 从什么情绪→中间经过什么→以什么情绪结束。至少三个节点，用箭头连接。不是"情绪变化"四个字 | 写"情绪起伏"——subagent 不知道从哪到哪 |

### 2. 情绪路径（不是直接跳变）

| 标准 | 判定方法 |
|------|---------|
| 至少三个节点 | mood_progression 必须有"从 A → 经过 B → 到 C"的结构。一个情绪直接跳到另一个情绪 = 不合格 |
| 节点之间有因果 | 不是"A 情绪→B 情绪→C 情绪"的罗列，而是"因为 XX 事件，情绪从 A 转向 B"——每个情绪转换必须有事件锚点 |
| 不能是所有场景同情绪 | 一整章只有一种情绪 = 扁平。即使是过渡章，也必须有情绪微调 |

### 3. 强度和弧线方向

| 字段 | 填写标准 | 常见错误 |
|------|---------|---------|
| intensity_peak | 指出哪一段情绪最强烈（具体到 key_point 或场景），不是"结尾"两个字 | 写"结尾"——subagent 不知道哪一段算结尾、以什么方式达到峰值 |
| intensity_level | 1-10 数字。相邻章强度差不超过 2（连续两章强度差过大 = 读者疲劳）。不是随便填 | 每章都填 7 或者每章都填 3——没有强弱变化，一卷下来情绪平摊 |
| emotional_hook | 章尾钩子的情绪类型（悬念/共情/期待/愤怒/好奇/选择/渴望/威胁）。不是"有意思" | 写"悬念"三个字——什么悬念？读者会对什么悬念？必须具体 |

### 4. micro_payoffs（微兑现）

每章至少 1 个微兑现——读者在本章能获得的小收获。过渡章也必须至少 1 个。没有微兑现的章 = 读者看完觉得"什么都没发生"。

| 字段 | 填写标准 | 常见错误 |
|------|---------|---------|
| type | 从 7 种中选：info / relationship / emotion / clue / ability / resource / recognition | 留空或写"待定" |
| description | 一句话描述读者获得了什么。必须是具体事件，不是"获得了信息" | 写"获得了线索"——什么线索？关于什么的？ |
| location | 前段/中段/后段——告诉 subagent 放在本章的哪个位置 | 不写或者写"任意"——subagent 可能把它放在章末，冲击了 emotional_hook |

```
✅ 合格：
primary_mood: "好奇"
mood_progression: "日常松弛→好奇心被勾起→悬念收束（带着问题结束）"
→ subagent 知道：开头松弛、中间勾起好奇、结尾落回悬念。三段清晰

intensity_peak: "苏沫描述朋友圈地址查无此址的瞬间——这是陆征真正被勾起兴趣的时刻"
→ subagent 知道：高潮在苏沫说"查无此址"的那一段，不需要猜

emotional_hook: "悬念（苏棠到底在哪？朋友圈地址为什么查无此址？）"
→ subagent 知道：章末要落在这个悬念上——读者想知道"苏棠去哪了"和"地址为什么不存在"

micro_payoffs:
  - type: "info"
    description: "读者获得核心悬念的基本信息：苏棠失踪两周，手机留家，警方结论与家属判断矛盾"
    location: "中段"
→ subagent 知道：中段必须让读者觉得自己掌握了案件基本信息——这是本章的小回报

❌ 不合格：
primary_mood: "复杂"
mood_progression: "先抑后扬"
→ "复杂"是什么？"先抑后扬"只有两个节点，没有中间过渡

intensity_peak: "结尾"
→ 哪一段结尾？什么场景？怎么达到峰值？全是空的

emotional_hook: "悬念"
→ 什么悬念？读者对什么产生悬念？subagent 不知道要落什么钩子

micro_payoffs: []  ← 空列表，无理由
→ 看完这章读者什么都得不到——这一章为什么存在？
```

## 三、hooks（钩子管理）

钩子的真相源在各 chapter.yaml 的 hooks 字段（不在全局 hooks.yaml）。每条钩子的生命周期（pending→mentioned→partial_advance→resolved）必须在各章的 memo.payoff_plan 中逐章推进。

### 1. 本章钩子操作

| 检查项 | 标准 |
|--------|------|
| payoff_plan 三个列表完整 | must_resolve / must_hold / partial_advance 都有值（空列表也必须明确写 `[]`，不能省略字段） |
| 每条钩子有操作理由 | 为什么本章必须收/必须压/只能推进——理由写在 advance 字段里 |
| 钩子 ID 可追溯 | payoff_plan 中的 hook ID 必须能在之前某章的 payoff_plan 或 chapter.yaml 中找到首次出现 |
| partial_advance 有推进边界 | advance 字段写的是"推进到什么程度"，不是"推进了"。边界不清 → subagent 可能推进过头 |

### 2. 跨章钩子追溯

| 检查项 | 标准 |
|--------|------|
| 每条活跃钩子在最近 3 章内至少被提及一次 | 超过 3 章未提及 → 钩子死了，应标记为 abandoned |
| 钩子状态不能倒退 | pending 可以变为 partial_advance 或 resolved，但不能变回 pending |
| 不能同时出现在 must_resolve 和 must_hold 中 | 逻辑矛盾，立即修正 |
| 不能收束从未埋过的钩子 | 如果 must_resolve 中出现新 hook_id，要么是之前漏埋了，要么是凭空收束 |

```
✅ 合格：
第 1 章：
  partial_advance:
    - id: "hook-001"
      advance: "种下主线钩子——苏棠失踪，警方判定主动离开，但细节有矛盾"

第 2 章：
  partial_advance:
    - id: "hook-001"
      advance: "勘查发现苏棠不是主动离开的证据；发现便利贴和被删聊天记录——指向有人知道更多"

第 3 章：
  partial_advance:
    - id: "hook-001"
      advance: "发现苏棠的失踪与三年前旧案的证人有关联——案件升维，从失踪案变为可能涉及灭口的刑事案件"
→ 三章都在推进 hook-001，每次推进都有具体的"推进到什么程度"。边界清晰，subagent 不会过度收束。

❌ 不合格：
第 3 章 payoff_plan：
  must_resolve: ["hook-005"]
  must_hold: ["hook-005"]
→ 同一条钩子既要收又要压——逻辑矛盾

第 5 章 payoff_plan：
  must_resolve: ["hook-010"]
→ 回溯前 4 章 chapter.yaml，hook-010 从未出现过——凭空收束，等于漏写了几章的钩子
```

## 四、AI 味自检（章纲阶段前置检查）

AI 味检查不是在正文写完后才做的——章纲阶段的 AI 味会直接污染提示词，然后被 subagent 抄进正文。章纲里的每一句"本章讲述了""主角经历了""通过 X 展现了 Y"都会变成正文里的上帝视角解构。

### 1. 无上帝视角解构

| # | 检测项 | 扫描字段 | 命中模式 | 修正方法 |
|---|--------|---------|---------|---------|
| 1 | 上帝视角概述 | outline.summary、outline.key_points、memo.current_task | "本章讲述了""主角经历了""本章将展示""读者将看到" | 改为"写 X 发生了 Y"——直接给事件，不包裹叙述层 |
| 2 | 抽象心理总结 | memo.reader_expectation、memo.key_choices | "他感到""他意识到""他内心挣扎" | 改为外部可观察的行为或信息："他合上笔记本，推门出去"比"他感到不安"可写 |
| 3 | 分析性结论 | memo.reader_expectation、memo.required_changes | "这表明""这意味着""这标志着关系转折" | 只写"什么变了"，不写"这意味着什么" |
| 4 | 文学评论腔 | memo.current_task、memo.downtime_functions | "通过X展现了Y""完成角色弧光转折""深化主题" | 改为"X 场景中 Y 做了 Z"——写事件，不写评论 |
| 5 | "不是而是"句式 | 所有 memo 字段 | "不是 X，而是 Y" | 直接写 Y。如果必须用对比，改为两个独立句。X → 一句话。Y → 一句话 |
| 6 | 抽象标签替代具体事件 | memo.current_task、memo.required_changes | "关系递进""冲突升级""性格成长" | 写"谁和谁的关系从 X 变成 Y"、"冲突从 X 升级为 Y"、"角色从不会 X 到会 X" |

### 2. memo 无空洞形容词评价

| 检测项 | 命中词 | 修正方法 |
|--------|--------|---------|
| 空洞形容词 | "精彩""紧张""有趣""不错""好""棒""优秀""震撼""感人""深刻" | 改为可验证的描述——删掉"紧张"，写"他在后视镜里看那人的手在口袋里" |
| 抽象副词 | "深刻地""充分地""淋漓尽致地""完美地" | 全部删除——副词没有信息量，subagent 无法执行 |
| 未量化的程度词 | "充分""足够""适当""一定程度" | 改为具体数量或范围——"描写 3 句"好于"适当描写" |

### 3. 每条 memo 可执行验证

| 检测项 | 判定方法 |
|--------|---------|
| current_task 可执行 | 读完这句，能说出"正文要写哪几个场景"吗？ |
| reader_expectation 可执行 | 读完这三行，知道"正文前半章给什么、后半章要什么"吗？ |
| payoff_plan 可执行 | 每条 partial_advance 的 advance 读完后，知道"这章该碰钩子的哪部分"吗？ |
| required_changes 可执行 | 每条的"从什么变成什么"读完后，知道"章末什么必须不一样"吗？ |
| prohibitions 可执行 | 每条的禁止项读完后，正文写完后能检查"这条违反了吗"吗？ |

```
✅ 合格：
memo.current_task: "通过现场勘查推进案件调查，线索逐渐指向苏棠不是主动离开的"
→ 无上帝视角概述、无抽象心理总结、无分析性结论、可执行（勘查→发现矛盾→推翻结论）

memo.required_changes:
  - "信息改变：陆征发现出租屋状态与'主动离开'矛盾——床没铺、充电器还在墙上、冰箱有剩饭"
  → 不写"这意味着案件复杂了"——只写发现了什么具体矛盾

❌ 不合格：
memo.current_task: "本章讲述陆征进入出租屋勘查，通过细致入微的观察展现了主角专业能力，使案件取得了重大突破"
→ "本章讲述" = 上帝视角概述；"细致入微" = 空洞副词；"展现了" = 文学评论腔；"重大突破" = 抽象标签

memo.required_changes:
  - "案件取得突破性进展，主角与反派的关系出现了微妙变化"
→ "突破性进展" = 什么进展？"微妙变化" = 什么变化？全是空词

outline.summary: "主角经历了一天紧张刺激的调查，最终发现了关键线索，这标志着他距离真相又近了一步"
→ "经历了" = 上帝视角概述；"紧张刺激" = 空洞形容词；"关键线索" = 不具体；"这标志着" = 分析性结论。整句报废。
```

## 章纲阶段自检

走完上面全部规则后，逐项过：

### memo 7 段
- [ ] **current_task** 可执行（能说出"本章要写哪几个场景"），不含抽象标签，不含上帝视角概述
- [ ] **reader_expectation** 的 state/strategy/detail 三个子字段都有值，detail 具体到人/事/物
- [ ] **payoff_plan** 三个列表完整（空列表显式 `[]`），每条 partial_advance 写了推进方式
- [ ] **downtime_functions** 每条有 `[场景] → [功能]` 映射，无功能的过渡段已删除
- [ ] **key_choices** 每条回答了"为什么做 + 符合人设吗 + 读者觉得突兀吗"
- [ ] **required_changes** 1-3 条，每条有改变类型 + "从什么变成什么"
- [ ] **prohibitions** 每条可验证（正文写完后能对照检查），不含抽象禁忌

### emotional_design
- [ ] **primary_mood** 是精确情绪词，不是"复杂""混合"
- [ ] **mood_progression** 至少三个节点（A→B→C），节点间有因果锚点
- [ ] **intensity_peak** 具体到 key_point 或场景，不是"结尾"
- [ ] **intensity_level** 与相邻章差不超过 2
- [ ] **emotional_hook** 有具体内容（谁/什么事/什么悬念），不是裸情绪类型词
- [ ] **micro_payoffs** 至少 1 个，type/description/location 三个字段都有值
- [ ] 不存在所有场景同情绪——即使是过渡章也有情绪微调

### hooks
- [ ] **payoff_plan 中每条钩子 ID 可追溯**到之前某章的 chapter.yaml
- [ ] **无 must_resolve 和 must_hold 冲突**（同一条钩子不能同时出现）
- [ ] **无凭空收束**（must_resolve 中的钩子在前文被埋过）
- [ ] **活跃钩子在最近 3 章内至少被提及一次**（超过 3 章未提及 → 标记 abandoned）
- [ ] **钩子状态不倒退**（resolved 不会变回 partial_advance）

### AI 味自检
- [ ] **outline 无上帝视角概述**（无"本章讲述了""主角经历了"）
- [ ] **memo 全部 7 段无分析性结论**（无"这表明""这意味着""标志着"）
- [ ] **memo 全部 7 段无文学评论腔**（无"通过X展现了Y""深化主题"）
- [ ] **memo 全部 7 段无"不是而是"句式**
- [ ] **memo 全部 7 段无空洞形容词**（精彩/紧张/有趣/不错/震撼/感人）
- [ ] **每条 required_change 是具体事件**，不是抽象标签（"关系递进"→不合格）
- [ ] **每条 memo 可执行验证**——对着 5 个判定问题逐条回答，全部答得出来

---

### 快速嗅探（30 秒内完成）

如果不想逐条过，先做这三个快速嗅探——三个全过，大概率合格：

1. **遮住章纲，能说出"这一章结束时什么东西变了"吗？** → 说不出来 → required_changes 没写清楚
2. **对着 reader_expectation.detail，能说出"正文前半章写什么、后半章要什么"吗？** → 说不出来 → memo 不可执行
3. **扫描 memo 全部 7 段，找到一句"本章讲述了""主角经历了""通过X展现了Y"吗？** → 找到了 → AI 味污染，立即修

---

### 验证方法

**优先级：Agent Teams > subAgent > 自检**

1. **Agent Teams（最优）**：开独立 reviewer agent，传入 `vol-{N}-ch-{M}.yaml` + 活跃角色 yaml + 前 3 章 chapter.yaml + 本清单。reviewer agent 逐项核查，对每项结论给出"通过/不通过 + 证据 + 具体位置"。
2. **subAgent（次优）**：同上，使用 subagent 执行审查。subagent 对 memo 7 段的"可执行性"判断能力弱于完整 agent。
3. **自检（兜底）**：严格按本清单逐项过。特别注意三个高危盲区：
   - **可执行性**：最容易目测放行。必须逐条反问"subagent 看到这句知道写哪段吗？"——答不出来就改。
   - **AI 味污染**：章纲写了 20 分钟后，人对自己的文字会脱敏。自检时建议把 memo 复制到纯文本里换字体再读——换字体可以打破脱敏。
   - **钩子追溯**：对着前 3 章 chapter.yaml 的 payoff_plan 逐条比对——看清楚每条钩子的状态演变，不允许跳跃或倒退。

**规则：拿到结论后先按 fail 项改完，再进入 Step 3（提示词生成）。不得带着 fail 项进入提示词生成。**

### 自检实战示例

以下是同一章纲"自检前"和"自检后"的对比——展示自检清单的实际作用。

**自检前（有 4 个致命问题）：**

```
# memo
current_task: "本章讲述主角调查案件，通过紧张的情节推进使读者感受到故事的节奏"
  → 问题1: "本章讲述" = 上帝视角概述
  → 问题2: "紧张的情节推进" = 空洞形容词
  → 问题3: "使读者感受到" = 分析性结论
  → 问题4: 不知道具体写什么场景——不可执行

reader_expectation:
  state: "期待"
  strategy: "维持悬念"
  detail: "读者期待下一章"
  → 三行全空——不知道读者在等什么、该给什么

required_changes:
  - "调查取得进展"
  → 什么进展？知道了什么新东西？等于没写
```

**自检后（修正为可执行的版本）：**

```
# memo
current_task: "陆征进入苏棠出租屋勘查现场。通过房间物品状态（床没铺/充电器还在墙上/冰箱剩饭），推翻警方'主动离开'的结论。同时在电脑回收站发现被删聊天记录——有人曾警告苏棠'别去那个地方'"

reader_expectation:
  state: "好奇——上一章结尾读者想知道苏棠去哪了"
  strategy: "制造新缺口"
  detail: "读者期待调查有进展——前半章通过房间矛盾细节给进展（床没铺=非主动离开），后半章通过聊天记录和楼下男人制造新疑问（谁在警告她？为什么有人盯梢？）"

required_changes:
  - "信息改变：确认苏棠非主动离开——房间状态3个矛盾细节"
  - "信息改变：发现被删聊天记录——发件人警告苏棠'别去那个地方'"
  - "威胁引入：陆征离开时被陌生男人盯上——调查已经触动某人"
```

每个 fail 项都对应一个修正——修正后的 memo，subagent 拿到就能写。
