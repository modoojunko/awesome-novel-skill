# AI 协作写小说？你可能一直在用错的方式

## 一、痛点：你和 AI 的协作为什么会卡壳？

你试过用 AI 写小说吗？大概率遇到过这些情况：

**写到后面忘了前面**
> "第三章主角说要复仇，第五章突然忘了这茬……"
> 人物状态前后矛盾，伏笔埋了没回收，读者一眼就看出断层。

**AI 味太重，越写越像套路文**
> "他瞳孔一缩，身体一僵，陷入了沉思……"
> 通篇都是这类描写，读起来像流水线产品，毫无灵魂。

**写着写着不知道下一步该写什么**
> 章与章之间没有因果链，每章结尾和下章开头对不上。
> 写到哪里算哪里，最后整个故事散成一盘沙。

**协作全靠"记忆力"**
> AI 记不住你设定的细节，你得一遍遍重复：
> "主角叫张远，武力值 80，不是 90！"
> "这个世界观没有飞机！"

这些问题几乎每个尝试 AI 协作写作的人都会遇到。

---

## 二、根因：不是 AI 不够聪明，是协作架构有缺陷

**问题的根源不在于 AI 模型，而在于「人与 AI 的协作方式」缺乏系统性设计。**

常见的 AI 写作模式是"问答式"：
- 你问："帮我写一段主角出场"
- AI 答："好的，这是一个英俊潇洒的主角……"
- 你继续问："再加一段打斗"
- AI 继续答……

这种模式的问题：
1. **没有全局上下文管理** — AI 每次只看到当前对话，前面的设定、人物状态全靠你重复灌输
2. **没有质量门禁** — 写出来就是写出来了，没有验收标准，没有 AI 味检测
3. **没有状态追踪** — 写到第 10 章忘了第 3 章埋的伏笔
4. **没有结构约束** — 想到哪写到哪，故事容易失控

结果是：协作效率低、产出质量差、写作体验崩。

---

## 三、方案：状态驱动的工作流系统

**awesome-novel** 是一个专为 AI 协作写作设计的工作流系统。

它的核心理念是：**用文件管理状态，用流程保证质量，用规范消除 AI 味。**

### 核心技术：状态驱动循环

```
检测状态 → 匹配路由 → 执行子任务 → 更新状态 → 回到检测
```

整个系统通过文件状态驱动：
- `chapter.md#status` 标记章进度：`outline` → `draft` → `archived`
- `.agent/status.md` 记录全局进度
- 角色状态、历史、情绪弧线全部落盘，不靠"记忆力"

每次启动，从磁盘读取状态，精确恢复到上次位置。

---

### 架构：主 Agent + 子 Skill 的分层设计

```
┌─────────────────────────────────────────────────────┐
│                   主 SKILL.md                        │
│  ┌─────────┐   ┌──────────┐   ┌────────────────┐  │
│  │ Step 1  │ → │ 决策信号表 │ → │ 路由分发        │  │
│  │ 检测状态 │   │(status+  │   │ → 对应子 Skill  │  │
│  │         │   │ 文件存在) │   │                │  │
│  └─────────┘   └──────────┘   └────────────────┘  │
└─────────────────────────────────────────────────────┘
                     │
         ┌───────────┼───────────┬────────────┐
         ▼           ▼           ▼            ▼
   ┌──────────┐ ┌─────────┐ ┌─────────┐ ┌──────────┐
   │  setup   │ │ outline │ │ chapter │ │  prompt  │
   │ 项目初始化 │ │ 主线拆纲 │ │ 章纲设定 │ │ 提示词生成│
   └──────────┘ └─────────┘ └─────────┘ └─────────┘
                                                  │
         ┌──────────┐ ┌─────────┐ ┌─────────┐  │
         │  review  │ │ archive │ │  write  │ ←┘
         │ 深度评审  │ │  归档   │ │ 正文生成 │
         └──────────┘ └─────────┘ └─────────┘
```

**关键设计：主 Agent 不读写项目文件**

| 角色 | 职责 |
|------|------|
| **主 Agent** | 状态检测 + 路由分发。只读 `status.md` 和 `chapter.md#status`，不直接操作项目文件 |
| **子 Skill** | 执行具体任务。读设定文件、写章纲/提示词/正文、更新状态 |

---

### 每个 Skill 的设计规范

每个子 Skill 严格遵循 **Harness Engineering** 的 7 字段契约：

| 字段 | 含义 | 示例 |
|------|------|------|
| **Role** | 一句话定义这个 Skill 是谁 | "全章正文写作 Agent" |
| **Scope** | 做什么 + 不做什么 + 边界 | "只读提示词文件，不读其他设定" |
| **Inputs** | 开始前必须读取的文件 | `prompts/vol-{N}-ch-{M}-prompt.md` |
| **Outputs** | 完成后写入的文件 | `archives/vol-{N}-ch-{M}-{slug}.draft.md` |
| **Tools** | 允许使用的工具 | `Read` / `Write` / `Bash`(仅文件重命名) |
| **Verification** | 什么算"做完了" | 字数 ≥ 目标 80%，文件存在 |
| **Lifecycle** | 开始读什么状态，结束留什么记录 | 读 `chapter.md#status=draft`，写 status→archived |

```
│ Inputs                                                 │
│ ├─ references/genre-style.md（题材指南）               │
│ ├─ references/world-setup-style.md（世界观指南）       │
│ └─ references/character-setting-style.md（角色指南）   │
│                                                        │
│ Outputs                                                │
│ ├─ story.md（项目索引）                                │
│ ├─ settings/writing-style.md（写作风格）               │
│ ├─ settings/genre-setting.md（题材设定）               │
│ ├─ settings/world-setting.md（世界观）                 │
│ └─ settings/character-setting/{id}.md（角色文件）      │
│                                                        │
│ 流程 SOP                                               │
│ 1. 执行 init.py 创建项目骨架                           │
│ 2. 题材选择 → 24种类型，选定后不动                     │
│ 3. 写作风格确认 → 默认低AI味/可调整                     │
│ 4. 类型档案选择 → 可选，预置类型知识库                │
│ 5. 题材配置（5字段）→ 满足类型/节奏/套路/禁忌         │
│ 6. 世界观（3项）→ 地理/政治/规则逐项聊                │

│                                                        │

│ └─ End: status.md current_phase=volume                 │

```

**解决痛点：协作全靠"记忆力"**
- 设定文件一次性写入磁盘，后续章节自动读取
- 角色认知6层模型确保人设不跑偏

#### 2. novel-outline — 主线拆纲 + 卷纲规划

```
┌────────────────────────────────────────────────────────┐
│ novel-outline                                          │
├────────────────────────────────────────────────────────┤
│ Role: 从结局倒推规划全书主线，再逐卷展开                │
│                                                        │
│ Scope                                                  │
│ ├─ 做: 主线拆纲/卷方向/章节列表                        │
│ └─ 不做: 逐章章纲/提示词/正文                          │
│                                                        │
│ Inputs                                                 │
│ ├─ story.md（项目索引）                                │
│ ├─ settings/world-setting.md                           │
│ └─ references/story-arc-style.md（主线指南）           │
│                                                        │
│ Outputs                                                │
│ ├─ story.md#story_arc（主线+分卷规划）                 │
│ └─ volumes/volume-{N}.md（卷纲+章节列表）              │
│                                                        │
│ 流程 SOP — 主线拆纲（仅首次）                          │
│ 1. 判断作者类型                                        │
│    ├─ A类：知道全书走向 → 走全流程                     │
│    └─ B类：只知道开头 → 只定第一卷，后续标"待定"       │
│ 2. 总主线 → 提取"谁+追求什么+对抗什么"                 │
│ 3. 定终点 → 从结局倒推，不问"几卷"问"最后一幕"       │
│ 4. 找断点 → 起点到终点的根本性变化                    │
│ 5. 定每卷冲突 → 一卷一卷确认，每卷服务于主线          │
│ 6. 三向核对 → 总主线→卷/卷序列→完整路径/卷→反推主线  │
│                                                        │
│ 流程 SOP — 卷方向确定                                  │
│ ├─ 卷1（首次）→ 展示故事结构模板（三幕/起承转合/悬疑） │
│ └─ 卷N+1（有前卷）→ 角色发声法，从角色意志推导方向    │
│                                                        │
│ 流程 SOP — 卷纲讨论                                    │
│ 1. 定核心冲突 → "谁+想做什么+被什么阻碍"              │
│ 2. 拆章节 → 每章三要素：谁做什么+冲突+结束时什么变了 │
│ 3. 检测4种不合格章纲：主题式/功能式/梗概式/结果式     │
│ 4. 章节因果链 → 前章章末变化是后章起点               │
│ 5. 卷级4验证 → 每章可追溯卷冲突/章末推动故事前进     │
│                                                        │
│ Verification: 15秒嗅探全过（核心冲突/章末变化/因果链）│
│                                                        │
│ Lifecycle                                              │
│ ├─ Start: status.md current_phase=volume               │
│ └─ End: status.md current_phase=chapter-loop           │
└────────────────────────────────────────────────────────┘
```

**解决痛点：章与章之间没有因果链**
- 从结局倒推确保每卷服务于主线
- 章纲必须含"结束时什么变了"，因果链可追溯

#### 3. novel-chapter — 章纲设定

```
┌────────────────────────────────────────────────────────┐
│ novel-chapter                                          │
├────────────────────────────────────────────────────────┤
│ Role: 从卷纲拆出每章详细规划，包含8段Memo+情绪设计    │
│                                                        │
│ Scope                                                  │
│ ├─ 做: 章方向/纲要点/Memo/情绪设计/Hooks              │
│ └─ 不做: 正文写作/提示词生成                           │
│                                                        │
│ Inputs                                                 │
│ ├─ volumes/volume-{N}.md（卷纲）                      │
│ ├─ chapters/vol-{N}-ch-{M-1}.md（上一章章纲）         │
│ └─ references/chapter-setting-style.md                │
│                                                        │
│ Outputs                                                │
│ └─ chapters/vol-{N}-ch-{M}.md                        │
│    ├─ status: outline                                  │
│    ├─ outline（方向+关键点）                          │
│    ├─ memo（8段式）                                   │
│    ├─ emotional_design（6字段）                       │
│    └─ hooks（钩子操作）                                │
│                                                        │
│ 流程 SOP                                               │
│ 1. 方向输入                                            │
│    ├─ 章级角色发声 → 处境/未完成的事/现在想做的事     │
│    ├─ 收集4组输入 → 读者缺口+角色状态+钩子+卷定位    │
│    └─ 合成1-3个方向选项，展示给作者确认               │
│ 2. 展开纲要点 → 字数倒推法+三段锚点法                 │
│ 3. 禁止清单 → "本章有没有明确禁止出现的元素？"        │
│ 4. Chapter Memo（8段）                                 │
│    ├─ current_task / reader_expectation               │
│    ├─ payoff_plan / downtime_functions               │
│    ├─ key_choices / knowledge_state                  │
│    └─ required_changes / prohibitions                │
│ 5. Emotional Design（6字段）                          │
│    ├─ primary_mood / mood_progression                │
│    ├─ intensity_peak / intensity_level               │
│    ├─ emotional_hook / micro_payoffs                 │
│    └─ micro_payoffs: 每章至少1个，类型6选1            │
│ 6. Hooks操作 → 埋/推进/收束，记录到本章hooks字段      │
│ 7. 验收（强制4步）                                     │
│    ├─ 给作者看 structured 反馈                        │
│    ├─ 检查清单24项自检                                │
│    ├─ 快速嗅探（能说出章内容吗？）                    │
│    └─ AI味自检（6大类检测）                           │
│                                                        │
│ Verification: 作者明确说"对"才算合格                  │
│                                                        │
│ Lifecycle                                              │
│ ├─ Start: chapter.md 不存在 → 新章                     │
│ └─ End: status=outline → 自动分发到 novel-prompt     │
└────────────────────────────────────────────────────────┘
```

**解决痛点：写到后面忘了前面**
- 角色发声确保每章从角色当前状态出发
- Hooks字段追踪伏笔生命周期，不遗漏

### 状态驱动循环的工作原理

```
┌─────────────────────────────────────────────────────────┐
│                    跨会话恢复流程                        │
├─────────────────────────────────────────────────────────┤
│                                                         │
│  用户: "继续写小说"                                      │
│         │                                               │
│         ▼                                               │
│  ┌─────────────────────────────────────────────────┐     │
│  │ Step 1: 检测状态                                 │     │
│  │ ├─ 读 .agent/status.md → current_phase/chapter │     │
│  │ ├─ 读 latest chapter.md#status → outline/draft  │     │
│  │ └─ Bash检查 files exist → prompt/archives       │     │
│  └─────────────────────────────────────────────────┘     │
│         │                                               │
│         ▼                                               │
│  ┌─────────────────────────────────────────────────┐     │
│  │ 信号表匹配                                       │     │
│  │ ├─ status=outline + 无prompt → novel-prompt    │     │
│  │ ├─ status=draft + 无draft → novel-write        │     │
│  │ ├─ status=draft + 有draft → body-verify        │     │
│  │ └─ status=archived → 下一章章纲                │     │
│  └─────────────────────────────────────────────────┘     │
│         │                                               │
│         ▼                                               │
│  ┌─────────────────────────────────────────────────┐     │
│  │ 精确恢复到上次位置                               │     │
│  │ ├─ 读 volume-{N}.md → 获取本章方向             │     │
│  │ ├─ 读角色文件 → 恢复当前状态                   │     │
│  │ └─ 从章纲/提示词/正文开始继续                   │     │
│  └─────────────────────────────────────────────────┘     │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

**SOLO模式特殊处理：**
- 每完成一章 → 告诉作者"输入/new继续下一章"
- 正文写作消耗大量上下文，新会话读文件恢复
- 归档不完整时禁止触发/new，确保磁盘状态完整

### 错误恢复机制

| 失败场景 | 恢复策略 |
|---------|---------|
| init.py不可用 | 手动创建目录+复制templates/ |
| subagent超时 | 主Agent降级直接写 |
| 文件被占用 | 等5秒重试，最多3次 |
| 状态不连续 | 报作者"第N章未归档，是否跳过？" |
| 角色状态不一致 | 退回archive修复再继续 |

---

### 工具契约（Tool Contract）

主Agent与子Skill的工具权限严格分离：

| 工具 | 主Agent | 子Skill |
|------|--------|--------|
| **Read** | story.md / status.md / SKILL.md / references | ✓ 读设定文件 |
| **Write/Edit** | status.md（仅此文件） | ✓ 写项目文件 |
| **Bash** | init.py / ls / grep | 仅文件重命名 |
| **Agent** | 启动子Skill | ❌ 禁止 |
| **WebSearch** | 作者明确要求时 | ❌ 禁止 |

**硬性禁止：**
- 主Agent禁止自行判断"不需要dispatch"
- 主Agent禁止直接读写项目文件（status.md除外）

## 四、流程与 SOP

### 完整写作循环

```
Phase 1: 设定（一次性）
├── 题材选择（24种类型 + 满足类型分析）
├── 写作风格（叙事身份/核心原则/常见错误/描写技法）
├── 世界观（地理/政治/规则三层次）
└── 角色设定（认知6层模型）

Phase 2: 卷纲（主线拆纲 → 卷展开）
├── 主线拆纲（从结局倒推法）
└── 卷纲设定（章节列表 + 因果链验证）

Phase 3: 逐章循环（可跨会话持久化）
├── 章纲设定（8段式 + 情绪设计6字段）
├── 提示词生成（6模块骨架）
├── 正文生成（独立写作子 Agent）
├── 正文验收（15项质量门禁）
├── 深度评审（10维60+细项诊断）
└── 归档（角色状态追加 + 状态更新）
```

### 关键 SOP 规范

**1. 世界观设定 — 三层次逐项聊**

```
地理：主角在哪 → 怎么去 → 路上什么样
政治：谁说了算 → 谁限制他 → 不服会怎样
规则：先问不能做什么 → 再问能做什么 → 代价是什么
```

**2. 角色设定 — 认知6层模型**

```
务虚层（为什么做）          务实层（怎么做）
├── 世界观（我是谁在世界的位置）
├── 我的定位（我对身边人的位置）
└── 价值观（什么必须坚持）   ├── 能力（能做什么）
                            ├── 技能（具体怎么做）
                            └── 环境（在什么条件下）
```

**3. 章纲设定 — 必须回答的问题清单**

1. 当前任务：主角这章要完成什么？
2. 读者期待：读者希望看到什么？
3. 兑现计划：前文铺垫怎么回收？
4. 过渡功能：这章在卷中的桥梁作用？
5. 关键抉择：主角必须做什么选择？
6. 角色信息状态：角色知道/不知道什么？
7. 章尾改变：这章结尾必须发生什么变化？
8. 硬约束：什么绝对不能出现？

**4. 正文验收 — 10维60+细项**

| 维度 | 细项 |
|------|------|
| 章纲执行度 | 22项 — memo/情绪设计/hooks 兑现检查 |
| 卷纲推进度 | 4项 — 因果链/节奏/张力 |
| 世界设定一致性 | 7项 — 地理/政治/规则不冲突 |
| 角色一致性 | 14项 — 能力/性格/决策模式连贯 |
| 写作风格执行度 | 22项 — 四字段锚定验证 |
| 前文延续性 | 5项 — 伏笔回收/状态衔接 |
| 钩子管理 | 8项 — 钩子强度/回收时机 |
| AI 味检测 | 6大类 — 副词/动词/连接词/身体反应/元叙事 |

**AI 味检测示例词**

| 类别 | 疲劳词 |
|------|--------|
| 副词类 | 忽然、突然、竟然、默默、渐渐 |
| 动词类 | 露出、勾起、泛起、浮现、陷入、涌起 |
| 连接词类 | 然而、不过、于是、随即、与此同时 |
| 元叙事类 | 本章讲述了、接下来、故事还在继续 |
