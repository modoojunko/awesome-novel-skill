# awesome-novel v4.6 → v4.8 视频脚本输入文档

> 用途：视频脚本素材。每节可独立成段，也可以串联成完整叙事。
> 风格：面向网文作者/编剧/写作爱好者，技术细节点到为止，重点讲"解决了什么"。

---

## 一、写小说到底难在哪

### 1.1 从想法到文字的断层
每个作者都有这种体验：脑子里画面很满——这个角色什么性格、这场戏要达成什么效果、这章情绪要怎么走——但落到键盘上就是写不出来。或者写出来了，读起来像两个纸片人在念台词。

**核心矛盾：** 作者知道"要发生什么"（剧情结构），但卡在"角色具体怎么演"（行为细节）。

### 1.2 场景描写无从下手
环境只会写"阳光洒进来"，外貌只会写"肤白貌美大长腿"，转场永远是"三天后"。不是不想写好，是缺乏一套可复用的方法论。

### 1.3 AI 写作的通病
用 AI 写小说最大的问题是"AI 味"——句子工整但没人味，角色行为模板化，缺乏真人作者的粗糙感和即兴感。

### 1.4 流程混乱
写到哪了、下一章该干嘛、之前埋的伏笔还记得吗——这些琐事消耗大量创作精力。

## 二、v4.6 → v4.8 解决了什么

### 2.1 【核心新特性】角色推演沙盘

**问题：** 有卷纲（知道这章要发生什么），但展开不到章纲（不知道角色具体怎么演）。

**什么是推演沙盘：**
它是一个交互式工具。作者设好"舞台"——场景在哪儿、谁在场、气氛什么样——再给一个触发事件，然后 AI 逐个角色推演：

```
第1轮：
  ├─ 角色张三（主角）：在愤怒的驱动下说了什么、做了什么、表情如何
  │     ↑ 作者可以修改张三的输出
  └─ 角色李四（反派）：看到张三的反应后，他怎么回应
        ↑ 作者可以修改李四的输出

第2轮：
  ├─ 张三：看到李四的回应后，进一步反应
  └─ 李四：……
```

关键设计：
- **串行推演**：一个角色演完才到下一个，因为人的行为是相互影响的
- **每轮可修改**：作者觉得"不对，这个角色不会这么说"，当场改，改完后下一个角色看到的是修改后的版本
- **多轮迭代**：直到剧情走到作者满意的位置
- **信息差透明**：AI 只使用该角色当前知道的信息，不知道的事不能表现出来

**产出物：** 一份推演记录，记录每个角色每轮的动作/台词/神态。作者拿着这份记录去写章纲，就不会卡了。

**触发条件：**
- 作者主动说"跑一下推演"/"剧情推演"
- 作者说"卡住了"/"写不下去"——AI 会主动建议用沙盘

### 2.2 scene-craft 方法论补全

**问题：** 场景写作缺乏可复用的方法论，每次都要从零想。

**解决方案：** 三个知识库文件，用标签化的方式组织写作技法。

#### 环境描写（三套方法）
- **五感通写法**：不只用视觉，叠加听觉/嗅觉/触觉。
  - 普通人："破庙脏乱，满是灰尘蛛网"
  - 高手："木门吱呀作响，殿内回荡回音，空气中飘着腐草与积尘气味，寒气顺着脚踝向上蔓延"
- **情绪渲染法**：景物随主角心境变化。
  - 心情好："阳光如金，野花向阳"
  - 心情差："阳光刺目，花草沾满污泥"
- **细节互动法**：靠器物侧面交代处境。
  - 不写"他家境贫寒"，写"搪瓷缸边缘磕出缺口，牙膏管被反复挤压至扁平"

#### 角色外貌描写（新增侧面烘托）
以前有"锚点法"（抓一个特征放大）、"动态法"（融入动作流程），新增：
- **侧面烘托法**：不直接描述长相，借旁人反应反衬。
  - 不写"她容貌绝美"
  - 写"她踏入大厅的瞬间，喧嚣声戛然而止。正举杯的客人悬着酒杯忘了饮"

#### 场景转场（七种技法）
- 情绪顺延、危机钩子、五感反差、物件线索、动线镜头、空镜景物、对话承接
- 每种技法都附带❌反面例子和✅正面例子

### 2.3 推演记录→作者→章纲（正确的流程）

这是 v4.8.0 开发过程中纠正的关键设计。

**错误设计（最初）：** 推演记录自动注入 prompt-crafter → AI 根据推演记录写正文

**正确设计（修正后）：** 推演记录给作者看 → 作者据此写章纲 → 章纲走正常流程 → 正文

原因很简单：推演记录是**创作素材**，不是**程序输入**。作者的创作决策不能被自动化替代。

## 三、完整写作流程（从设定到归档）

### 3.1 项目初始化
```
作者说"帮我写本小说"
  ↓
AI 聊世界观/角色/风格/题材
  ↓
写入设定文件
```

### 3.2 卷纲规划
```
四维方法论同时推进：
  ① 情绪走向（读者读这卷情绪怎么变化）
  ② 冲突阶梯（2-4 层逐级升高的障碍）
  ③ 信息差（谁知道什么、谁不知道什么）
  ④ 场景卡（每章三要素）
```

### 3.3 章纲规划
```
卷纲有章节核心故事（一句话）
  │
  ├─ [正常] → 直接展开章纲（情绪/冲突/信息差/场景卡）
  │
  └─ [卡住] → 推演沙盘 → 推演记录 → 作者参考 → 写章纲
```

### 3.4 正文写作
```
章纲 → prompt-crafter 组装提示词 → writer 写正文
  → anti-ai 去 AI 味 → (可选) reader 评审 → 归档
```

每一步都由专门的 AI agent 负责，novel-agent 像项目经理一样调度。

### 3.5 归档与维护
归档时自动做：
- 更新角色状态
- 追踪伏笔/钩子健康度
- 检查节奏（连续高压超过3章或连续平淡超过2章，提醒作者）
- 卷边界检测

## 四、这个系统跟其他 AI 写作工具有什么不同

### 4.1 不是"AI 替你写"，而是"AI 帮你写"
大部分 AI 写作工具是：你给一段提示 → AI 吐一段文字 → 你改 → 再吐。

awesome-novel 的设计哲学是：AI 负责执行层面的工作（方法论注入、格式校验、节奏检测），创作决策始终是作者的。

### 4.2 推演沙盘 vs 直接让 AI 写
直接让 AI 写："帮我写章陆征和方岩对峙的章纲"→ AI 吐一段文字 → 不合适 → 重来 → …

推演沙盘：你先看角色怎么演 → 你觉得哪个反应不对 → 当场改 → 下一个角色基于修改继续演 → 你看到全部互动后再决定章纲怎么写。

区别在于：推演沙盘让你**看到过程**，而不是只拿到结果。

### 4.3 知识库不是死的
scene-craft 的标签化设计意味着 AI 会根据当前场景自动匹配合适的写作技法，而不是每次都把整篇知识库塞进提示词。稀疏抽取——每个场景类型只选 1-2 条。

## 七、视频脚本可用的话术片段

> "你有没有这种经历？脑子里画面很饱满——这个场景什么样、谁在说什么、情绪怎么走——但坐到电脑前就是写不出来。"

> "其实卡文不是你的问题，是缺少一个'把脑子里的画面变成文字'的工具。"

> "推演沙盘就像导演在排练现场：你告诉演员这场戏要演什么，然后让每个演员走一遍，你看哪个地方不对，喊停，让他们重来。"

> "我们的推演记录不是给 AI 看的，是给你看的。你拿着这份记录去写章纲，就像导演拿着排练记录去写拍摄脚本。"

> "不是所有 AI 写作工具都应该'替你写'——有时候你最需要的是有人陪你一起想。"

> "scene-craft，意思就是场景工艺。环境怎么写、外貌怎么写、转场怎么接——每一件都有方法，不用从零硬想。"
