# Self-Checklists Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Create 5 comprehensive self-checklist files in `references/`, update install.sh and SKILL.md to wire them into the workflow.

**Architecture:** Each checklist is a standalone reference document in `references/`, following WVP's SCRIPT-STYLE.md format: bottom lines → principles with examples/anti-examples → final checklist → verification method. The `install.sh` copies `references/` alongside `skills/` and `scripts/`. SKILL.md's 硬性自检协议 section references each file by path.

**Tech Stack:** Markdown reference docs, shell script (install.sh)

# 章纲自检清单

> **两条底线**（任一不过，禁止生成提示词）：
>
> 1. **memo 7 段完整**——悬念、节奏、情绪、视角、对话、描写、落点，缺一段都不行。缺 = subagent 自行脑补 = 放飞。
> 2. **情绪设计有弧线**——本章不是平铺直叙。必须有情绪起点→变化→落点。哪怕是过渡章，情绪也在流动。

---

## 最高原则：章纲是写给 AI 的需求文档

章纲不是给作者看的备忘——是给写作 subagent 的唯一指令。subagent 没有直觉，ta 只知道你写下来的东西。
所以：**章纲里写"这一章要有悬念感" = 没写。章纲里写"读者在第 3 段之后才知道那是陷阱" = 写了。**

---

## 一、memo 7 段填写标准

### 1. 悬念

**合格：** 读者在这一章结束时想知道什么？具体到"想知道 X 是不是 Y"。
**不合格：** "这一章要有悬念" / "留下悬念"（悬念什么？没说等于没写）

例：
- ❌ "本章末尾留个悬念"
- ✅ "本章末尾让读者以为主角死了，下一章开头再揭示是假死。具体手法：第 5 段描写主角被淹没前的手势特写，第 7 段水面恢复平静。"

### 2. 节奏

**合格：** 本章的快慢分配。哪一段要紧迫、哪一段可以缓。
**不合格：** "节奏紧凑"（太抽象，subagent 不知道怎么执行）

例：
- ❌ "本章节奏先松后紧"
- ✅ "前三段日常对话（舒缓），中间打斗开始（加速），第 5-7 段连续冲突（高强度），结尾 1 段回落（余韵）"

### 3. 情绪

**合格：** 读者读这一章应该感受到什么情绪？情绪起点和终点在哪？
**不合格：** "温馨"（没头没尾）

例：
- ❌ "情绪基调：悲伤"
- ✅ "情绪起点是和解后的宁静，中段因发现新线索转为紧张，落点在不安的预感"

### 4. 视角

**合格：** 本章用谁的视角写的？视角是否稳定？如果切换了，切换点在哪里？
**不合格：** "第三人称"（这是废话——你要说是"紧贴主角的第三人称有限"还是"上帝视角"）

例：
- ❌ "第三人称"
- ✅ "紧贴主角的第三人称有限视角。读者只知道主角知道的信息。第 4 段切换 2 句到反派的办公室（仅此一次），其余全部跟随主角。"

### 5. 对话

**合格：** 本章对话的功能——推进剧情、暴露角色、还是制造冲突？谁和谁之间？
**不合格：** "有对话"（当然有，问题是为什么有）

例：
- ❌ "本章有 A 和 B 的对话"
- ✅ "A 和 B 在废墟上对峙。B 试图说服 A 交出地图，A 用沉默拒绝。对话功能：暴露 B 的真正目的 + 展示 A 的固执。"

### 6. 描写

**合格：** 本章需要重点描写的环境/物品/角色外貌，及其叙事功能。
**不合格：** "描写一下 X"（为什么描写？为了凑字数？）

例：
- ❌ "描写一下城市废墟"
- ✅ "重点描写城市废墟中的图书馆——烧焦的书架散落在地，暗示文明已崩塌。这处环境的叙事功能：让主角意识到回不去了。"

### 7. 落点

**合格：** 本章结束在哪？读者读完最后一句是什么感受？
**不合格：** "主角继续前进"（等于没结束）

例：
- ❌ "主角决定去找真相"
- ✅ "本章结尾在主角站在悬崖边——前是深渊，后是追兵。读者感受：绝望 + 想知道怎么脱身。"

---

## 二、情绪设计（emotional_design）

- [ ] 本章的情绪起点明确
- [ ] 本章的情绪落点明确
- [ ] 情绪路径合理（不是直接跳变——有过渡）
- [ ] 强度和弧线方向已标注（上升/回落/持平/转折）

**常见错误：**
- ❌ 平铺直叙：没有情绪起伏，像在念说明书
- ❌ 为了转折而转折：前一秒还在笑后一秒就哭，没有铺垫
- ✅ 情绪变化有触发事件：角色是因为经历了什么才变化

---

## 三、hooks

- [ ] 本章铺设的钩子已标注（伏笔/暗示/未解释的现象）
- [ ] 本章收束的钩子已标注（回应的前文伏笔）
- [ ] 钩子有具体触发条件（不是一个笼统的"怀疑"）
- [ ] 如果收束了之前的钩子——钩子在 chapter.yaml 的 hooks 字段中有相同 id

---

## 四、AI 味自检（章纲阶段前置检查）

- [ ] 章纲中没有"上帝视角"的解构——subagent 写正文时会直接抄进去
- [ ] memo 没有使用空洞的形容词评价（"好""不错""精彩"）
- [ ] memo 的每一条都是**可执行的指令**——subagent 拿到后知道具体写什么
- [ ] 没有在 memo 里写"按照 XXX 风格"——具体规则在提示词里，不在章纲里

---

## 章纲自检

- [ ] **memo 7 段全部有值，且每段可执行**（subagent 拿到知道写什么）
- [ ] **情绪设计完整**（起点→路径→落点）
- [ ] **hooks 已标注**（铺设/收束 + 条件）
- [ ] **AI 味自检通过**（无上帝视角、无空洞评价、可执行）
- [ ] **章节概要不是"主角做了 X"——而是"读者应该感受到什么"**

### 验证方法

同上 Teams > subAgent > 自检。

特别注意：拿一条 memo 出来，问自己"subagent 看到这个，知道具体写哪一段、达成什么效果吗？" 如果 memo 只能给人类看而不能给 AI 用 → 重写。

# 提示词自检清单

> **两条底线**（任一不过，禁止进入正文生成）：
>
> 1. **视角已转换**——章纲是上帝视角，提示词必须锁定视角。没转换就直接喂给 subagent = subagent 交替使用不同视角 = 读者困惑。
> 2. **writing-style 四字段已注入**——role / core_principles / possible_mistakes / depiction_techniques 缺一个。缺 = subagent 在无约束状态下写正文 = 放飞。

---

## 最高原则：提示词是 subagent 的宪法

subagent 不会质疑你写了什么。ta 会执行每一个字。
所以：提示词里写"注意不要写成 XXX" = subagent 会在脑子里想一遍 XXX。**提示词只应该说你想要的，不要用否定来引导。**

---

## 一、视角转换

### 转换规则

章纲里写的"主角发现 X / 反派在计划 Y / 读者感受到 Z" 在提示词中必须转换成：

章纲："主角走进废墟，发现了一封信。读者应该感受到不安。"
提示词转换：
- ❌ "主角走进废墟发现一封信，内心感到不安"（上帝视角旁白）
- ✅ "走进废墟。脚下是烧焦的木屑。门框斜挂着。桌上放着一封信——信封角落有三道血痕。"（读者通过主角的感官发现信，不安感来自环境细节，不靠说）

**检查清单：**
- [ ] 没有"主角感到""主角意识到""读者应该感受到"这样的心理断言
- [ ] 所有心理活动通过动作/对话/环境来暗示
- [ ] 视角标记明确（"紧贴 X 的视角，读者只知道 X 知道的信息"）
- [ ] 如果在章纲里写了多角色视角，已确认本章锁定单一视角（除非明确的切换点）

### 叙事段落 function 分配

- [ ] 每个 segment 有明确的叙事功能（atmosphere / dialogue_push / action_beat / emotional_landing / transition）
- [ ] 功能分配合理（不是全章都是 action_beat，也不是全章 atmosphere）
- [ ] segment 之间过渡自然（不是生硬切割）

---

## 二、writing-style 注入

### 四字段确认

- [ ] **role** — AI 的角色定位（"你是 X 风格的写手，擅长 X"）
- [ ] **core_principles** — 本章必须遵守的核心原则（已从 writing-style.yaml 提取并自然语言重述）
- [ ] **possible_mistakes** — 本章易犯的错误列表（已提取并自然语言重述）
- [ ] **depiction_techniques** — 推荐的描写技法参考（已按场景类型对应注入）

### 类型注入

- [ ] 已从 genre-corpus 读取 prompt_segment（如适用）
- [ ] 已按场景类型注入对应的 few-shot 示例（如有 style-extract 产出）
- [ ] genre_taboos 已作为约束写入（不是禁令列表，是"不写什么"的指引）

---

## 三、章纲 → 提示词完整性

- [ ] 本章概要已转化为提示词的开头上下文（不是整段粘贴）
- [ ] 前一章结尾的状态已说明（角色在哪、什么情绪、接下来要做什么）
- [ ] 本章落点已说明（subagent 知道在哪里收笔）
- [ ] 字数目标已设定（不是"写多少算多少"）

---

## 提示词自检

- [ ] **视角已转换**（无上帝视角心理断言）
- [ ] **叙事 function 分配合理**（segment 各有功能）
- [ ] **writing-style 四字段已注入**
- [ ] **类型 prompt_segment 已追加**（如有）
- [ ] **few-shot 示例已注入**（如有 style-extract）
- [ ] **提示词只说你想要的**（没有用否定句式来引导）
- [ ] **subagent 看到提示词后不需要猜测任何事**

### 验证方法

用人话"跑一遍"提示词——想象自己是 subagent，读完后：
- 知道这一章写什么吗？
- 知道从谁的视角写吗？
- 知道在哪里结束吗？
- 知道什么风格/什么不能写吗？

如果有"这里可能需要猜一下"的感觉 → 补全再进正文生成。

# 正文自检清单

> **两条底线**（任一不过，禁止归档）：
>
> 1. **AI 味检测通过**——疲劳词命中数 ≤ 阈值，句式违规数 ≤ 阈值。超标 = 必须在当前章节改，不能"下一章注意"。
> 2. **章纲承诺全部兑现**——memo 7 段的每一段承诺在正文中都有对应。章纲说"读者会感受到不安"，正文里没有不安 → 章纲和正文脱节。

---

## 最高原则：正文服务于 memo

memo 是本章的合同。subagent 写的正文如果不满足 memo 中的任何一段 -> 合同未履行 -> 改。
这也是为什么 memo 必须可执行、必须写清楚——memo 如果有歧义，正文出问题不是 subagent 的错，是章纲的错。

---

## 一、AI 味检测

### 1. 疲劳词检测

扫描正文，按 anti-ai.yaml 的 blocklist 分类逐项检查：

- [ ] 副词类疲劳词（忽然、突然、竟然、居然、默默、微微、渐渐、逐渐、依旧、仍然）
- [ ] 动词类疲劳词（露出、勾起、泛起、浮现、陷入、涌起、充斥）
- [ ] 形容词类疲劳词（复杂、微妙、难以言喻、莫名、异样、浓重）
- [ ] 连接词类疲劳词（然而、不过、但是、却、于是、随即、紧接着、与此同时）
- [ ] 身体反应类疲劳词（瞳孔一缩、身体一僵、倒吸一口凉气、呼吸一滞）

**规则：** 同义词替换优先于直接删除。如果删掉后句子意思缺失，说明这个词有功能，换成更具体的表达。

**示例：**
- ❌ "他忽然感到一阵莫名的恐惧"（忽然 + 感到 + 莫名 = 三重疲劳）
- ✅ "后颈凉了凉。他回头——走廊空无一人。"（具体感受替代抽象情绪）

### 2. 句式违规检测

按 tic-patterns.yaml 的 8 种句式模式逐项扫描：

- [ ] **"不是……而是"句式**——"他不是害怕，而是愤怒" → "他在愤怒。"
- [ ] **身体部位情绪模板**——"心脏狠狠跳动了一下" → 直接写身体反应
- [ ] **"仿佛/像是"堆砌**——"天空像是被撕裂了一样" → "天空裂开。"
- [ ] **"随着"开头**——"随着脚步声靠近，他的心跳加速" → "脚步声靠近。心跳加速。"
- [ ] **"让/令/使"句式**——"这一幕让他想起昨天" → "他想起了昨天。"
- [ ] **"当"句式**——"当他走进房间时，所有人看着他" → "走进房间。所有人看着他。"
- [ ] **"了"的滥用**——每段不超过 3 个"了"
- [ ] **感知词掩盖**——"看到/听到/闻到/感到"后用抽象描述

### 3. 对话规则检测

- [ ] 对话标签自然（"说/道/问"为主，不是副词标签"他愤怒地说"）
- [ ] 对话之间有动作/环境作为节奏间隔（不是"他说""她说"连续交替）
- [ ] 不使用"!"来替代情绪描写

---

## 二、15 项质量检查

### 叙事完整性

- [ ] **memo 兑现**——7 段中的每一段在正文中有对应体现
- [ ] **章节落点符合**——正文结束位置与章纲规划的落点一致
- [ ] **情绪弧线匹配**——读者的情绪路径（起点→变化→落点）与 emotional_design 一致

### 技术质量

- [ ] **视角一致**——没有在段落中间悄悄切换视角
- [ ] **时态一致**（如适用）
- [ ] **人称一致**（没有混用"他"和"你"或"我"）
- [ ] **段落长度合理**（没有连续 5 段一样的长度）

### 内容质量

- [ ] **没有"本书""本作"类元叙述**——角色不会说"我在这部小说里"
- [ ] **没有预设读者反应的句子**（"读者可能已经注意到""正如前面提到的"）
- [ ] **具体信息准确**（角色名字、地名、力量体系名称前后一致）
